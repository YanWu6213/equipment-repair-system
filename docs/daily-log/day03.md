# Day 3｜接上資料庫、完成基礎 CRUD 與 EF Core 遷移

## 今天做了什麼
- 解決 .NET 8 與 EF Core 10 的相容性問題，手動將 NuGet 套件版本切換到 8.0.x。
- 建立 `AppDbContext`，讓程式具備與資料庫溝通的能力。
- 使用 `Add-Migration` 建立遷移，讓 C# 類別自動對應成資料庫表格。
- 成功實作 POST（新增）功能，讓資料可以真正寫入資料庫並保留下來。
- 補齊 GET（全部/單筆）、POST（新增）、DELETE（刪除）功能，完成基礎 CRUD 流程。
- 將資料庫相關操作改成 `async / await`，讓 API 寫法更符合實務開發模式。

## 今天完成了什麼
- 專案已成功接上資料庫，不再只是用記憶體假資料。
- 已能透過 Migration 建立資料表，而不是手動建表。
- 已完成資料的新增、查詢與刪除。
- API 的資料流程已具備基本的持久化能力。

## 遇到的問題
- 安裝 EF Core 套件時出現 `NU1202`，原因是套件版本與專案使用的 .NET 8 不相容。
- 一開始尚未執行 `Update-Database`，導致資料庫與程式碼不同步，程式無法正確運作。
- 測試 API 時曾出現 404 與 500，主要是因為路由、參數傳遞或測試方式不夠清楚。

## 我最後怎麼解
- 將 EF Core 套件版本手動調整回 8.x，與 .NET 8 對齊。
- 透過套件管理員主控台執行 Migration 與資料庫更新指令，讓資料表成功建立。
- 使用 Swagger 直接測試 API，透過修改 JSON 與傳遞 ID 參數來驗證單筆查詢與刪除功能。

## 今天學到的關鍵觀念
- `DbContext` 是程式與資料庫之間的核心橋樑，負責管理資料存取。
- `Migration` 的概念不是手動建表，而是讓程式碼作為資料庫結構的來源，也就是 Code First 的核心精神。
- 相依注入（Dependency Injection）可以讓 Controller 在建構子中直接取得資料庫物件，而不用自己手動建立。
- `async / await` 可以讓程式在等待資料庫或磁碟 I/O 時不阻塞主流程，提升 API 的回應效率。

## 今天的反思
- 我今天最大的進展，是讓 API 從「假資料展示」正式進化到「真資料存取」。
- 我今天最需要注意的地方，是套件版本和專案框架必須對齊，否則一開始就會卡在安裝階段。
- 我今天更明白，資料庫開發不是只會寫 SQL，而是要理解程式、資料表與遷移之間的關係。

## 下次我可以少踩什麼坑
- 安裝 NuGet 套件前，先確認自己的 .NET 版本與套件主版本是否一致。
- 在 EF Core 中，執行 `Add` 或 `Remove` 後，要記得呼叫 `SaveChangesAsync()`，不然資料不會真正寫入。
- 寫 API 時要更有意識地使用正確的 HTTP 狀態碼，例如 `NotFound()`、`NoContent()`，讓回應更符合 RESTful API 習慣。
