# Day 2｜EquipmentRepairSystem：從單筆資料進階到搜尋與路由設計

## 今天做了什麼
- 學會使用 `List<Ticket>` 作為資料容器，模擬從資料庫抓出多筆報修單的情境。
- 成功讓 API 從原本只能回傳單一物件，進階到回傳整個 JSON 陣列。
- 實作 `SearchTickets` 功能，讓使用者可以透過網址參數（Query String）搜尋特定維修員的工單。
- 學會使用自定義路由，例如 `[HttpGet("all")]`，來區分不同的 API 入口。

## 今天完成了什麼
- API 已能回傳多筆資料，而不只是單一物件。
- 已能透過查詢參數完成簡單搜尋功能。
- 已理解並修正 API 路由衝突問題。
- Swagger 能再次正常辨識並載入 API。

## 遇到的問題
- 在新增方法時漏掉 `[HttpGet]`，導致 Swagger 無法辨識該 API。
- 一度把教學說明用的標記內容誤貼成程式碼，導致編譯器出現 `CS7014` 錯誤。
- 兩個 API 方法都使用預設路徑，造成「動詞 + 路徑」重複，導致系統啟動時出現路由衝突，Swagger 也跟著無法載入。

## 我最後怎麼解
- 養成每個 API 方法上方都明確標註 HTTP 動詞的習慣，例如 `[HttpGet]`。
- 移除所有不屬於 C# 語法的文字或標記，恢復程式碼純淨。
- 在 `[HttpGet("all")]`、`[HttpGet("search")]` 中加入自訂路徑，手動分流不同 API 的網址。

## 今天學到的關鍵觀念
- `LINQ` 的 `Where` 可以搭配 Lambda 寫法 `t => t.Property`，像過濾網一樣，從多筆資料中挑出符合條件的內容。
- 後端每一支 API 的「HTTP 動詞 + 路徑」組合必須唯一，否則系統會發生路由衝突。
- 當 C# 回傳 `List` 時，前端看到的是被 `[]` 包起來的 JSON Array；若是單一物件，則會是 `{}`。

## 今天的反思
- 我今天最大的進展，是從「只會回傳單一資料」進步到「可以回傳多筆資料並做簡單搜尋」。
- 我今天最容易犯的錯，是沒有先規劃 API 路徑，就直接新增方法，結果讓兩支 API 撞路。
- 我今天更明白，後端開發不只是把功能寫出來，還要設計清楚每個 API 的入口。

## 下次我可以少踩什麼坑
- 複製範例時，先確認哪些是解說文字、哪些才是真正的程式碼。
- 在新增方法前，先想好 API 的網址命名，例如 `/all`、`/search`，避免路由衝突。
- 養成先看 Error List 再按 F5 的習慣，提早發現問題。
