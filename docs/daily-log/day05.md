# Day 5｜加入資料驗證與 DTO，讓 API 更安全

## 今天做了什麼
- 在 `Ticket.cs` 中加入 `Data Annotations` 驗證規則，例如 `[Required]` 與 `[RegularExpression]`。
- 透過 Swagger 測試，確認當 `status` 輸入不符合規則時，系統會自動回傳 `400 Bad Request` 與錯誤訊息。
- 建立 `TicketCreateDto`，正式引入 DTO（Data Transfer Object）模式。
- 讓 API 的輸入格式與資料庫模型分離，避免使用者直接操作不該填寫的欄位，例如 `id`。

## 今天完成了什麼
- API 已具備基本的資料合法性檢查能力。
- 成功驗證後端可以自動攔截不符合規則的資料。
- 已開始把 API 的輸入模型與資料庫模型拆開。
- 專案結構從單純 CRUD，進一步走向比較實務的分層設計。

## 遇到的問題
- 一開始不確定類別是否一定要寫在 `namespace` 大括號內，擔心影響其他檔案抓取類別。
- 第一次面對多個類別檔案時，對 `DTOs` 資料夾應該放在哪裡、怎麼整理專案結構有些不確定。

## 我最後怎麼解
- 遵循命名空間規範，讓所有 Model 類別都正確放在對應的 `namespace` 中。
- 在專案根目錄下建立 `DTOs` 資料夾，並透過 `using` 引用不同資料夾中的類別，整理出更清楚的專案結構。

## 今天學到的關鍵觀念
- 後端不能直接信任外部傳入的資料，驗證是保護系統與資料品質的第一道防線。
- DTO 的價值不只是讓程式碼更整齊，更重要的是防止 Over-posting，並讓 API 介面與資料庫模型解耦。
- `400 Bad Request` 並不代表程式壞掉，而是後端正確執行了資料檢查，並回應前端「你的請求格式或內容不合法」。

## 今天的反思
- 我今天最大的進展，是讓 API 不再只是能接收資料，而是開始知道哪些資料該收、哪些不該收。
- 我今天最重要的理解，是 API 設計不能只看功能有沒有跑通，還要考慮安全性與資料邊界。
- 我今天更明白，專案結構一旦開始成長，就必須用資料夾與命名空間去整理，不然後面會越來越亂。

## 下次我可以少踩什麼坑
- 之後開發新功能時，可以先想前端真正需要傳哪些欄位，再先建立 DTO，避免 Controller 越寫越亂。
- 可以多利用 file-scoped namespace 讓程式碼更精簡、可讀性更高。
- 未來在設計 API 時，要更早思考哪些欄位應該由系統管理，哪些欄位才應該開放給使用者輸入。

## 明天要做什麼
- 繼續整理目前 API 的輸入與輸出模型，看看哪些地方還適合補 DTO。
- 檢查目前 CRUD 流程中，還有哪些欄位缺少驗證規則。
- 開始思考登入、使用者角色，或是 README / 專案文件的下一步整理。
