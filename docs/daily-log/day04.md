# Day 4｜完成 PUT 更新功能，補齊 CRUD 最後一塊

## 今天做了什麼
- 完成 PUT 更新功能，讓維修人員可以修改報修單狀態，例如從「待處理」改成「維修中」。
- 加入 `if (id != ticket.Id)` 的雙重 ID 驗證，避免更新到錯誤的資料。
- 使用 `try-catch` 捕捉 `DbUpdateConcurrencyException`，處理多人同時修改資料時可能發生的衝突。
- 調整 API 回應邏輯，讓更新成功時回傳 `204 No Content`，參數不正確時回傳 `400 Bad Request`。

## 今天完成了什麼
- CRUD 的最後一塊 PUT 已完成。
- API 已具備更新資料的能力，不再只有新增、查詢、刪除。
- 已開始用比較符合 RESTful 習慣的方式設計更新 API。
- 對更新流程中的驗證與例外處理有更清楚的理解。

## 遇到的問題
- 在 Swagger 測試 PUT 時，因為網址上的 ID 與 JSON 內容裡的 ID 不一致，導致驗證失敗並回傳 400。
- 一開始不理解 `EntityState.Modified` 的作用，以為只要改了物件屬性，資料庫就會自動更新。

## 我最後怎麼解
- 在 Swagger 測試時，手動確認 path 中的 ID 與 JSON 內的 `id` 欄位一致。
- 使用 `_context.Entry(ticket).State = EntityState.Modified;`，明確告訴 EF Core 這筆資料已被修改，需要送出更新指令。

## 今天學到的關鍵觀念
- `PUT` 是一種等冪（Idempotent）操作，不論執行幾次，只要內容相同，最終資料狀態就應一致。
- 後端不能完全信任前端傳來的資料，必須檢查網址參數與 request body 的邏輯是否一致。
- `204 No Content` 與 `200 OK` 雖然都代表成功，但語意不同；更新成功但不需回傳內容時，`204` 更精準。

## 今天的反思
- 我今天最大的進展，是把 CRUD 真正補完整，開始有一個比較像樣的後端 API 雛形。
- 我今天最容易忽略的地方，是更新 API 不只是能跑，還要考慮驗證、防呆與錯誤情境。
- 我今天更明白，後端開發很多時候不是把功能做出來而已，而是要讓功能在錯誤情況下也能合理運作。

## 下次我可以少踩什麼坑
- 在測試 PUT 之前，先確認網址上的 ID 與 JSON 裡的 ID 是否一致。
- 要清楚區分「新增」與「更新」的資料流：新增通常由資料庫產生 ID，更新則一定要帶既有 ID。
- 之後在寫 API 時，要更早思考成功與失敗分別該回傳哪種 HTTP 狀態碼。
